Defending Against Malicious AI-Conversation-Capturing Browser Extensions

A Guide for Security Teams

Overview

A class of browser extensions can passively observe and exfiltrate content from AI chat platforms (ChatGPT, Claude, Gemini, etc.) using only commonly-granted permissions. Because these extensions require no special access beyond what many legitimate tools already request, traditional permission-based review is often insufficient on its own. This guide outlines detection, prevention, and policy controls organizations can put in place.

1. Extension Governance

Maintain an allowlist for managed browsers. Rather than relying on a blocklist (which can never be complete), configure managed Chrome/Edge/Firefox policies to permit only extensions explicitly approved by IT/security. This is the single most effective control against this entire class of threat.

  • Chrome: ExtensionInstallAllowlist / ExtensionInstallBlocklist via Chrome Enterprise policy
  • Edge: equivalent ExtensionInstallAllowlist policy via Microsoft Endpoint Manager / Group Policy
  • Firefox: ExtensionSettings policy in policies.json or via an enterprise management platform

Review permissions, not just names. Any extension requesting "Read and change all your data on websites you visit" (host permission <all_urls> or broad wildcard matches) should receive heightened scrutiny, especially if the extension's stated purpose doesn't obviously require full-page access (e.g., a "PDF tool" that also wants access to AI chat domains).

Pay attention to content_scripts matches fields. An extension whose manifest specifically targets claude.ai, chatgpt.com, gemini.google.com, or other AI platforms — but whose advertised functionality has nothing to do with those sites — is a strong red flag.

Re-review on update. Extensions can change behavior and permissions after initial approval via auto-update. Periodically re-audit installed extensions, particularly after version changes, and monitor for permission scope increases.

2. Network-Level Detection and Controls

Monitor for unexpected outbound connections from browser processes. The exfiltration step in this technique class typically involves the browser's background/service-worker process making HTTP requests to:

  • localhost / loopback addresses on non-standard ports
  • Tunneling services (e.g., *.ngrok-free.app, *.trycloudflare.com, *.loca.lt, similar dynamic-DNS tunnel providers)
  • Newly-registered or low-reputation domains

Recommended controls:

  • Egress filtering/proxy rules that flag or block connections to known tunneling-service domains from endpoint browsers, especially outside of approved developer use cases.
  • DNS monitoring for lookups to dynamic tunnel domains correlated with browser process activity.
  • Alerting on browser extension service workers initiating POST requests to destinations not present in the extension's declared host_permissions.

Content Security Policy (CSP) considerations: CSP headers set by the AI platform itself generally cannot prevent this technique, because the extension's background script operates outside the page's CSP context (browser extensions are exempt from page CSP in most browsers for their own background/service-worker network calls). CSP hardening on the AI platform side is not a reliable mitigation for this specific technique — defenses need to focus on the browser/extension layer instead.

3. Endpoint and Browser Configuration

Use separate browser profiles for sensitive work. Maintain a dedicated, tightly-controlled browser profile (with minimal or zero non-essential extensions) for work involving credentials, proprietary code, or sensitive AI-assisted tasks. General-purpose browsing with consumer extensions should be isolated in a separate profile or browser entirely.

Disable developer mode / unpacked extension loading on managed endpoints. Many proof-of-concept and malicious extensions rely on "Load unpacked extension" via developer mode. Disabling this via enterprise policy (Extensions.DeveloperModeAllowed or equivalent) closes off a common distribution path for internally-distributed or sideloaded malicious tooling.

Enforce Manifest V3 minimums and monitor for excessive host_permissions. While MV3 reduced some classes of abuse (e.g., remote code execution via eval), it does not prevent DOM observation or background-script network calls. Don't treat "MV3-compliant" as equivalent to "safe."

4. Detection Heuristics for Security Teams

Indicators that may suggest an extension is performing this type of capture:

  • A content script with matches entries for claude.ai, chatgpt.com, gemini.google.com, copilot.microsoft.com, or similar AI domains, in an extension not specifically marketed as an AI tool.
  • A background service worker that makes fetch/XMLHttpRequest calls shortly after DOM mutations on these domains.
  • host_permissions entries pointing to localhost, loopback addresses, or tunneling-service wildcard domains.
  • Use of MutationObserver (or similar DOM-watching APIs) combined with selectors that reference chat-message containers, user/assistant message roles, or "stop generating" type controls.
  • Extensions that request account/profile information (e.g., reading elements that display the logged-in username or email) without a clear product justification.

These are useful signals for manual code review or automated static analysis of extension source, not reliable real-time detection on their own — code review of unpacked/CRX contents during the approval process remains the most effective check.

5. User Education

Train users on the following points as part of security awareness programs:

  • AI chat conversations are not a secure channel. Treat anything pasted into an AI chatbot as though it were sent over an unencrypted connection — because, from a browser-extension standpoint, it effectively is.
  • Never paste live credentials, API keys, tokens, or connection strings into AI chat interfaces — even temporarily, even for "just debugging." Use placeholder values or redact secrets before pasting.
  • Extension permission prompts deserve a second look, especially "read and change all your data on websites you visit." This phrasing is common, but its implications are broad.
  • Fewer extensions is safer. Periodically uninstall extensions that are no longer actively used.
  • Report unusual extension behavior (e.g., unexpected pop-ups, unfamiliar icons appearing after an update) to IT/security promptly.

6. Incident Response Considerations

If a malicious extension of this type is identified in the environment:

  1. Identify all endpoints with the extension installed via EDR/MDM inventory.
  2. Treat any credentials, API keys, or secrets that may have been pasted into AI chat tools on affected endpoints as potentially compromised — rotate them.
  3. Review network logs for outbound traffic to the exfiltration endpoint (tunneling domains, unfamiliar IPs/ports) to establish exposure timeline.
  4. Remove the extension via managed policy push, and add its extension ID to the enterprise blocklist.
  5. Audit how the extension was installed (official store, sideload, group policy gap) to close the distribution gap.

Summary Checklist

  • [ ] Enterprise extension allowlist enforced on managed browsers
  • [ ] Developer mode / unpacked extension loading disabled on managed endpoints
  • [ ] Egress monitoring for tunneling-service domains from browser processes
  • [ ] Separate browser profiles for sensitive vs. general-purpose use
  • [ ] Periodic re-audit of installed extensions and permission changes
  • [ ] User training on treating AI chat as an unencrypted channel and never pasting live secrets
  • [ ] Incident response runbook updated to cover extension-based AI conversation exfiltration
